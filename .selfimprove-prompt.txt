Implement this feature: Add Analytics & Error Tracking

The Browserbud team is operating completely blind in production. There is no analytics provider (PostHog, Google Analytics, Segment, etc.) and no error tracking service (Sentry, Bugsnag, etc.) installed on browserbud.com. This means: (1) there is zero visibility into user behavior — no data on which features are used, where users drop off, or what drives retention/churn; (2) production JavaScript errors and unhandled promise rejections go completely undetected until a user complains or silently churns. Every future product decision is a guess, and every production bug is invisible until it causes damage.

Solution: Integrate PostHog for product analytics and Sentry for error tracking into the Browserbud Next.js application. PostHog will be initialized in the root layout/app component using next/script with strategy='lazyOnload' to minimize performance impact, and will automatically capture page views, sessions, and custom interaction events. Sentry will be configured with both client-side and server-side configs to capture JavaScript errors, unhandled promise rejections, and API route errors, with source maps uploaded at build time (privately, not publicly exposed). All API keys will be stored as environment variables and never committed to the repository. A .env.example file will document required variables for future contributors. GDPR compliance will be addressed by respecting PostHog's opt-out mechanism and adding a minimal cookie consent notice for EU users.

Acceptance Criteria:
PostHog receives a `$pageview` event within 3 seconds of a user loading any page on browserbud.com, visible in the PostHog Live Events dashboard
PostHog receives a `$pageview` event on every client-side route navigation (not just initial page load), verified by navigating between pages and checking PostHog Live Events
Sentry captures and displays JavaScript errors thrown in the browser within 60 seconds of occurrence, verified by triggering a test error via browser console: `throw new Error('Sentry test')`
Sentry captures and displays unhandled promise rejections, verified by running `Promise.reject(new Error('async test'))` in the browser console on production
Sentry captures errors thrown in Next.js API routes (server-side), verified by adding a temporary `throw new Error('server test')` to an API route and confirming it appears in Sentry
All PostHog and Sentry API keys are stored exclusively in environment variables — running `git grep -r 'phc_\|sentry.io' -- '*.ts' '*.tsx' '*.js'` returns zero hardcoded secrets
The `.env.example` file exists and documents all required environment variable keys with placeholder values and explanatory comments
Lighthouse Performance score on the homepage does not decrease by more than 2 points compared to the pre-integration baseline (run Lighthouse before and after in the same environment)
PostHog script loads with `strategy='lazyOnload'` (or equivalent deferred loading), confirmed by inspecting the Network tab — PostHog JS must not block the critical rendering path
A cookie consent banner is displayed to first-time visitors and has not yet accepted cookies — PostHog must NOT fire any events until the user clicks 'Accept'
After clicking 'Accept' on the cookie banner, PostHog begins capturing events and the banner does not reappear on subsequent page loads
After clicking 'Decline' on the cookie banner, `posthog.has_opted_out_capturing()` returns `true` and no PostHog network requests are made
Sentry source maps are uploaded at build time but NOT publicly accessible — requesting a `.map` file URL directly in the browser returns a 404
No Sentry events are generated by automated bot traffic — the `beforeSend` hook filters requests where the User-Agent matches common bot patterns (Googlebot, etc.)
The `SENTRY_AUTH_TOKEN` is not present in `.env.local` or any committed file — it is only configured as a CI/CD environment secret
The global error boundary (`app/global-error.tsx`) renders a user-friendly fallback UI when a React render error occurs, and the error is captured in Sentry
